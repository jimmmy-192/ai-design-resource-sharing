# AI 设计资源精选

为初级 B 端设计师准备简洁的中文周度资源分享。默认精选 5 项已经核验的资源，解释每项资源的具体亮点，并让筛选过程可查看、可复核。

## 适用范围

服务对象：

- 0-3 年经验的 B 端产品或 UI 设计师。
- 正在探索 AI Coding、Vibe Coding 或 AI 全栈设计的设计师。
- 团队内部资源分享的读者。

使用简单易懂的中文。优先呈现具体亮点和有用的参考，不要求每项资源都能直接迁移到 B 端工作场景。

## 确定本期资源组合

每期可以混合不同类别的资源，不要求围绕单一主题，也不要求覆盖全部类别。优先组成一组既有启发性又有实用性的 5 项资源，例如同时包含视觉案例、设计提效站点、提示词资源和动效实现代码。

从以下资源类别中自由组合：

1. B 端复杂页面与流程设计。
2. C 端交互与视觉设计案例。
3. AI Coding 与 AI 设计工作流。
4. 动效、图标、提示词、可复用代码和设计提效站点。
5. 视觉表现突出或实现方式新颖的设计案例。
6. Vibe Coding 公开案例与 GitHub 项目。
7. 与设计师相关的 AI 新工具、产品更新和行业变化。

用户未指定重点时，直接建立多类别候选池，不询问用户选择单一主题。用户指定重点时，提高相关类别的比例，但仍可收录其他类别中明显优秀的资源。

## 质量优先的研究节奏

不以尽快找到 5 项为目标，也不要因为已经找到足够数量就提前停止核验。先完成候选池，再比较来源、素材完整度、亮点具体性和团队分享价值。第一轮候选整体偏弱时，继续扩展检索词、来源和相邻类别，直到能够解释最终项目为什么明显优于被淘汰项目。

不要为了凑够数量降低硬性门槛。经过扩展搜索后仍不足 5 项时，宁可少交并明确说明缺口，也不要加入证据不足、重复或质量一般的资源。

## 各类资源的寻找与筛选边界

根据候选资源所属类别使用对应标准。以下内容用于限定寻找方向、判断准确性和入选边界；同一期可以根据分享目标组合多个类别。

### B 端复杂页面与流程设计

优先选择能从界面图、案例视觉稿、原型或演示中观察到多步骤任务、信息层级、高密度布局、筛选、表格、状态反馈、权限或异常处理的案例。指出值得研究的具体流程或界面细节；不以真实线上交互、项目成果或完整设计过程作为入选前提。

### C 端交互与视觉设计案例

优先选择具有可观察的交互机制、反馈节奏、动效处理、导航模式或视觉构图的案例。高质量的界面图、案例视觉稿或动效视频本身即可作为交付素材；实际交互、项目成果和完整设计过程均非入选前提。聚焦案例本身的亮点，不强行要求提供 B 端迁移分析。

### AI Coding 与 AI 设计工作流

优先选择展示真实输入、过程和结果的资源。区分已经演示的能力与营销承诺，说明设计师可以尝试工作流的哪一部分。

### 动效、图标、提示词、可复用代码和设计提效站点

优先选择提供预览、示例、下载、试用或复用说明的资源。设计提效站点应能明确帮助设计师完成检索、生成、整理、协作、交付或开发中的至少一个环节，并提供可查看的功能、案例或实际使用入口。在重要时说明许可、价格、登录要求或访问限制。排除只有文字介绍、无法查看实际能力的合集。

### 视觉表现与新颖实现案例

可收录视觉完成度显著突出或实现方式新颖的作品，即使它不以复杂流程、可下载资源、真实项目成果或完整设计过程见长。视觉表现重点观察布局、信息层级、色彩、字体、图形、组件、材质、三维效果、动效和内容呈现；实现方式重点观察交互机制、转场、实时渲染、生成式视觉、技术组合或新的页面表达方式。

此类作品可以作为资源组合中的补充项，不得仅因不属于其他类别而直接淘汰。必须通过可查看的画面、视频、Demo 或实现说明指出具体亮点，不能只用“好看”“有创意”或“实现新颖”作为入选理由。

### Vibe Coding 公开案例与 GitHub 项目

优先选择结果可查看、源码或提示词可阅读、项目目的清晰且有近期活跃证据的项目。若以视觉案例形式收录，清晰的成品页面截图或演示视频即可，不以真实线上指标作为门槛。对于 GitHub，核验当前 Star 数、最近提交或 Release、README 质量，以及 Demo 或截图。

### AI 设计资讯

使用官方公告、更新日志或一手演示。只收录会实质改变设计师工作流或可用能力的更新。

## 按类型判断新鲜度

按资源类型使用对应的时间规则：

- AI 资讯和产品更新：30 天内发布或更新。
- 工具、仓库、素材和资源站：6 个月内存在发布、更新、提交或维护信号。
- B 端复杂页面与流程案例、以产品流程或交互机制为主要价值的 C 端案例：优先选择当前自然年度发布或更新的项目；符合下方“长期优秀案例”条件时，可以放宽年份。
- 视觉表现与新颖实现案例、Vibe Coding 案例：必须在当前自然年度公开发布、更新，或在案例页、作品集元数据中明确标注为当前年度；缺少精确月份不影响收录。

使用案例页、作品集、官方公告或仓库中的发布日期、更新日期、项目年份等元数据判断。不得使用截图、原型或界面中的订单日期、日志时间、数据记录等演示内容判断作品新鲜度；这些内容可能是随手填写的 Demo 数据。不要仅从视觉风格推断资源是否新鲜。

### 按期次判断时间范围

每期周报必须记录明确的时间范围，并按 7 天滚动：

- 第一期时间范围为用户指定的起止日期，例如 `2026-07-08 至 2026-07-15`。
- 下一期从上一期结束日开始，结束日顺延 7 天，例如第二期为 `2026-07-15 至 2026-07-22`。
- 用于筛选和去重时，按“包含开始日、不包含结束日”判断，避免边界日期重复收录；展示给读者时可以写成 `7.8-7.15`。
- 资源的发布日期、更新日期或维护信号优先落在本期时间范围内；不在本期范围内但仍符合“按类型判断新鲜度”的资源，可以作为补充候选，但需要在筛选记录中说明原因。
- 不确定本期起止日期时，先读取 `roundups/index.md` 中最近一期的时间范围，用最近一期结束日作为新一期开始日；如果历史中没有时间范围，再向用户确认第一期起止日期。

### 长期优秀案例例外

此例外只适用于“B 端复杂页面与流程设计”和以产品流程或交互机制为主要价值的“C 端交互设计案例”。C 端项目如果主要因为视觉效果突出而入选，应归入“视觉表现与新颖实现案例”，不得使用此例外。较早发布且长期未更新的案例同时满足以下条件时，仍可作为补充项呈现：

1. 原始案例页或创作者主页仍可访问，并能核验实际发布日期或项目年份。
2. 有成组且清晰的界面、流程、原型或案例素材，不是单张效果图或只有文字说明。
3. 可以指出今天仍然成立的具体价值，例如复杂任务拆解、信息层级、状态设计、反馈机制或交互流程。
4. 在交付中明确标记“经典案例”和实际年份，不得写成“近期发布”或“近期更新”。

默认每期最多收录 1 项长期优秀案例。此例外不适用于“视觉表现与新颖实现案例”、动效与图标资源、提示词、可复用代码、设计提效站点、工具、仓库、Vibe Coding 项目和 AI 资讯；这些类别仍执行原有新鲜度规则。

## 建立并评审候选池

在筛选最终 5 项前，先建立 12-15 项候选池。

以下任一硬性门槛不通过时，立即淘汰：

- 没有可核验的数据来源、官方网站、创作者主页或代码仓库。
- 链接失效，或实际内容无法访问。
- 没有足以观察和判断的视觉或可用素材，例如清晰界面图、成组案例视觉稿、动效视频、Demo、可下载素材或可运行项目。真实交互、项目成果和完整设计过程不是硬性要求。
- 不符合对应类型的新鲜度规则，且不满足“长期优秀案例”例外。
- 内容只有营销话术、基础入门教育，或属于无法追溯来源的转载。若营销类网站的视觉表现显著突出或实现方式新颖，则按“视觉表现与新颖实现案例”重新评估，不因其营销属性直接淘汰。

对通过硬性门槛的每个候选进行评分，再展示候选短名单：

| 维度 | 分数 | 判断标准 |
| --- | --- | --- |
| 受众与分享价值 | 0-3 | 对目标设计师有明确的启发、提效、复用或行业认知价值。 |
| 亮点具体性 | 0-3 | 能从视觉或可用素材中识别明确的界面、信息层级、视觉、动效、工作流或可复用能力。 |
| 实用价值 | 0-3 | 读者可以立即查看、收藏、研究、尝试、下载或复用；视觉研究价值也可得高分。 |
| 来源可信度 | 0-2 | 使用一手、可访问且有明确作品归属或案例信息的来源。 |
| 新鲜度 | 0-2 | 有清晰证据满足对应时间规则；长期优秀案例符合例外时记 1 分，不得记满分。 |
| 组合差异性 | 0-2 | 为本期提供不同于更高分候选的视角。 |
| **AI 评分** | **0-15** | 只展示 AI 评分不低于 10 分的候选。 |
| **你的评分** | **0-5** | 邀请用户按个人兴趣和团队匹配度评分；不得自行编造。 |
| **最终评分** | **0-20** | 用户 review 后，将 AI 评分与用户评分相加。 |

用户可参与 review 时，在最终确定前展示 6-10 个最强候选的表格。包含名称、类型、原始链接、一句话亮点、AI 评分、留空的用户评分，以及对接近入选却被淘汰候选的一句说明。

用户给出评分后，按最终评分排序。分数相同时，优先比较来源可信度和组合差异性。用户未评分时，明确说明本期仅基于 AI 评分筛选，不得虚构个人偏好。

## 正确使用发现来源

进行资源搜集时，先读取 [references/source-pool.md](references/source-pool.md) 中的长期发现来源池，再根据本期资源组合补充站外搜索。发现来源只用于建立每期候选池，不是最终引用来源。

建立候选池时，优先检查来源池中标注为“第一梯队”的周更合集。它们用于快速发现视觉案例、创意开发和交互作品；仍必须回到原始创作者页面、项目页或官方网站核验素材、实现信息与时间证据，不能把合集页直接当作最终引用。

当搜索在来源池之外的新网站上发现候选时，按以下规则维护来源池：

1. 先判断该网站是否是可以持续发现多个资源的平台、合集、社区、榜单或仓库集合。单个产品官网、创作者主页或单独项目页不加入长期来源池。
2. 只有该网站至少产出一个通过本 skill 硬性门槛的候选后，才将它加入 `references/source-pool.md`。候选不必最终入选，但必须有原始来源、可观察素材和有效的新鲜度证据。
3. 记录网站名称、网址、适合发现的类别、加入日期、本次通过门槛的示例，以及访问或内容质量限制。不要只记录一个裸链接。
4. 新来源加入后，当前期的 12-15 项候选仍记录在周报研究过程或历史中；不要把单期候选项目写进长期来源池。
5. 如果当前环境无权修改 skill 目录，在本期周报的“筛选记录”中写入“待加入来源池”，并在交付时明确说明，不得声称已完成更新。

每当用户觉得某个案例好时，先判断它是从合集网站中发现的还是单独站点：如果是通过合集网站发现的，可以收藏该网站；如果是单独站点，则先查看是否有对应的合集网站，只有在存在相应合集来源时才收藏，避免盲目收藏。

最终入选项链接到官方网站、原始创作者主页、原始案例页、项目页或 GitHub 仓库。不要机械限制数量，但要严格控制重复性；如果 a 和 b 讲的大部分是同一件事，这种情况下只保留一个。
## 撰写周报

筛选完成后，使用以下结构：

```markdown
# 本周设计资源精选

[用一句话说明本期值得关注的原因]

## 1. [资源名称]

- 类型：[案例 / 工具 / 动效资源 / 图标资源 / 提示词 / 可复用代码 / 设计提效站点 / 视觉案例 / 创新实现 / GitHub 项目 / AI 更新]
- 原始链接：[链接]
- 图片示例：[图片链接 / 截图说明]
- 时间状态：[近期发布 / 近期更新 / 持续维护 / 经典案例（实际年份）]
- 它是什么：[一句话说明]
- 亮点：[具体描述一个或两个界面、视觉、交互、流程或工作流细节]
- 为什么值得关注：[说明它对本期读者有何价值]
- 可以立刻怎么用：[给出一个明确行动，例如保存某种布局、研究一张界面图、尝试、下载或复用]
- 注意：[只在付费、语言、访问限制或使用门槛重要时说明]

## 2. [资源名称]

[重复到第 5 项]

## 本期观察

[用一到两句话总结这批资源体现的共同趋势]
```

不得使用“设计感强”“值得收藏”等没有具体理由的表述。

## 维护周报历史

使用项目级历史，不将真实周报记录写进 skill 的规则或参考资料：

- 在研究前读取 `roundups/index.md`，避免重复推荐近期资源或形成过于相似的资源组合。
- 为每期已定稿周报创建 `roundups/YYYY-Www.md`。
- 记录发布日期、本期时间范围、最终 5 项资源及其类别、来源域名、AI 评分、已有的用户评分，以及一句反馈或后续备注。
- 每期定稿后，在 `roundups/index.md` 追加一条简洁记录。

不要一次性读取全部历史。只读取索引和最近最相关的一两期周报。

## 最终检查

交付前确认：

- 已选择 5 项不重复资源；允许混合不同类别，不强制设置统一主题。
- 每项资源都通过硬性门槛，并有可访问的原始链接。
- 每个时效性描述都根据对应类型的新鲜度规则完成核验。
- 较早案例只有在满足“长期优秀案例”条件时才入选，并明确标记实际年份；视觉表现、设计提效和技术类资源未使用此例外。
- 每项资源都有足以支撑判断的视觉或可用素材；不把真实交互、项目成果或完整过程作为必需证明。
- 对视觉案例，只使用案例页或作品集元数据判断年份，未使用界面演示数据作为时间证据。
- 每项推荐都有具体亮点和下一步行动。
- 用户评分被如实记录，未给出的评分保持为空。
- 中文表达简洁清晰，适合初级设计师阅读。

## 基于真实使用迭代

已建立 `references/source-pool.md` 保存可持续扩充的发现来源。只在进行资源搜集、核验来源覆盖或更新来源池时读取它；不要因为 skill 被调用就机械读取无关参考资料。后续只有在多期真实使用中形成稳定的检索词、来源质量规律或团队认可写法时，才增加其他按需读取的参考文件。
